可见 kshell 即为 lshell，仅 Python 解释器路径不同 - /bin/kshell 使用的是 #!/usr/bin/env python，这表示它会根据当前系统的环境变量 PATH 来寻找 Python 解释器。 - /usr/bin/lshell 使用的是 #!/usr/bin/python，它会直接使用固定路径下的 Python 解释器。

3.3. iptables

防火墙规则

INPUT 链 (进入流量规则)
- policy ACCEPT：表示默认策略是 ACCEPT，即没有匹配规则的流量将被接受。
- 规则解释： - 第一条规则：丢弃所有进入服务器、目标端口为 4444 的 TCP 流量。
- 第二条规则：丢弃所有进入服务器、目标端口范围为 1337 到 x11（通常 6000-6063 的 TCP 流量，用于 X11 图形界面）之间的流量。
- 第三条规则：丢弃所有进入服务器、目标端口为 webmin 到 31337（webmin 默认端口为 10000，31337 通常是黑客工具的端口）之间的 TCP 流量。
- 第四条规则：丢弃所有进入服务器、目标端口为 webcache（通常指端口 8080）的 TCP 流量。
FORWARD 链 (转发流量规则)
- policy ACCEPT：表示默认策略是 ACCEPT。没有规则设置，因此所有转发的流量都被允许通过。一般在不做路由或转发情况下，这个链不会被使用。

OUTPUT 链 (输出流量规则) - policy ACCEPT：默认策略是 ACCEPT，即没有匹配规则的流量将被允许通过。 - 规则解释： - 第一条规则：丢弃所有从服务器发出的目标端口为 4444 的 TCP 流量。 - 第二条规则：丢弃所有从服务器发出的目标端口范围为 1337 到 X11（6000-6063）的 TCP 流量。 - 第三条规则：丢弃所有从服务器发出的目标端口为 webmin 到 31337 的 TCP 流量。 - 第四条规则：丢弃所有从服务器发出的目标端口为 webcache（8080）的 TCP 流量。 - 第五条规则：丢弃所有从服务器发出的目标端口为 www（80）的 TCP 流量，通常用于 HTTP。 - 第六条规则：丢弃所有从服务器发出的目标端口为 ftp（21）的 TCP 流量。

3.4. LFI

审计 member.php 存在 LFI【$page 参数】

首先，仅匹配 etc 做替换，而且仅替换 1 次，/etcetc/ 即可绕过

  • /etc/：这是正则表达式，用来匹配字符串中的 etc。
    • 这里 etc 是一个普通字符串，它没有使用正则表达式的特殊字符，所以它会直接匹配字符串中的 etc 片段。
    • / 是正则表达式的分隔符。
  • ''：这是替换后的字符串。在这个例子中是空字符串，所以匹配到的 etc 将被移除。
  • $page：这是要进行替换的目标字符串。该字符串可能包含 etc，并且 preg_replace 会在其中查找和替换。
  • 1：这是 preg_replace() 中的第四个参数，表示最多替换一次。如果字符串中有多个 etc，只有第一个匹配到的会被替换。

其次，利用 include 引入了 $memPage 文件，此处考虑截断即可，经测可利用 %00 成功截断

注，%00 截断文件包含利用条件： - magic_quotes_gpc=off - PHP版本低于5.3.4

结合 SQL 注入可实现 webshell 上传，反弹 shell等操作【SQL 注入引入php代码，文件包含触发执行】

3.5. UDF 提权

Using SQL Injection to perform SSRF/XSPA attacks
This has its own set of restrictions though. Based on the version of MySQL, which you can identify with select @@version, the directory where plugins can be loaded from is restricted. MySQL below v5.0.67 allowed for library files to be loaded from system path if the plugin_dir variable was not set. This has changed now and newer versions have the plugin_dir variable set to something like /usr/lib/mysql/plugin/, which is usually owned by root.
Basically for you to load a custom library into MySQL and call a function from the loaded library via SQL Injection, you would need the
- ability to write to the location specified in @@plugin_dir via SQL Injection
- file_priv set to Y in mysql.user for the current database user
- secure_file_priv set to "" so that you can read the raw bytes of the library from an arbitrary location like the network or a file uploads directory in a web application.

根据利用条件进数据库查配置【mysql -uroot】，当前用户已有最高权限

确认 FILE 权限

  • FILE 权限允许用户通过 SQL 语句来读取或写入服务器文件系统上的文件。
  • 'Y'：表示用户有 FILE 权限，可以执行类似 LOAD DATA INFILE 和 SELECT INTO OUTFILE 的操作。

查 SECURE_FILE_PRIV 得 "" ，即未限制导入|导出目录

SECURE_FILE_PRIV：这是 MySQL 的一个系统变量，用于限制 LOAD DATA INFILE、SELECT INTO OUTFILE 和其他与文件操作相关的 SQL 语句。它指定了一个目录，MySQL 只能在该目录内进行文件读取或写入操作。这样可以防止用户通过 MySQL 进行任意文件的读写操作，从而增强安全性。

  • 指定路径（例如 /var/lib/mysql-files/）：表示文件操作被限制在该指定目录。
  • NULL：表示文件操作被完全禁用，无法进行文件的读写。
  • "")：表示没有限制，文件操作可以在服务器的任何目录中进行（不推荐用于生产环境中，安全风险较大）。
  • mysql 5.6.34 版本以后 secure_file_priv 的值默认为NULL。并且无法用SQL语句对其进行修改。

查询插件目录发现未设置，但 mysql 版本低于 v5.0.67 ，也就是允许从系统路径直接加载库文件
